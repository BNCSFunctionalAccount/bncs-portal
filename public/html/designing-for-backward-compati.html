Designing for Backward Compatibility

This driver development kit differs significantly in approach & architecture to previous frameworks.  Drivers developed using previous frameworks or new drivers that need to support both BNCS V5.0 (and earlier) and V5.5 (and later) will require careful design to minimize the amount of implementation effort required to support both environments.

 

Our recommended approach is to place all device communication and driver implementation logic into a common library that can then be hosted inside both a V5.0 framework & V5.5 toolkit based implementation of the driver.  In this way, only code required to "wire up" the common library to each of the frameworks need be duplicated and the behaviour and performance of the driver should be consistent between implementations.

 

Since the V5.0 driver framework is based on the DotNet Framework (most recently V4.8) and the V5.5 driver development kit is based on DotNet Core, this common library should be developed in DotNet Standard 2.0 which is compatible with both.

 

The recommended approach for migrating a V5.0 Driver Framework driver to the V5.5 kit is to first separate out device communication code and logic into an external library, modify the V5.0 driver to link it to this library and then build a new V5.5 driver implementation within the same solution.
